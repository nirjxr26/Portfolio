## Overview

Bastion (internal codename AegisMesh) is a self-hosted, zero-trust IAM platform: authentication, policy-based authorization (PBAC), session management, audit logging, and ML-driven anomaly scoring, all running on a single-node K3s cluster under ArgoCD.

It's a Turborepo monorepo — three apps and two shared packages:

| Component | Stack |
|---|---|
| `apps/api` | Node.js / Express |
| `apps/dashboard` | React 18 / Vite |
| `apps/security-engine` | Python / FastAPI |
| `packages/shared`, `packages/ui` | TypeScript, React primitives |
| Database | PostgreSQL 17 via Prisma |
| Deployment | K3s (single node) + ArgoCD GitOps |

What follows walks each subsystem in the order you'd actually build it: the API, the database underneath it, the auth system built on top of both, the ML risk engine bolted onto auth, the dashboard that consumes all of it, then the platform — Kubernetes, CI/CD, observability — that runs the whole thing.

## The API Layer

The Express API (port 5000) is the single backend for every IAM operation: registration, login, JWT issuance and rotation, TOTP MFA, password reset, OAuth2 federation (Google/GitHub), PBAC checks, CRUD for roles, policies, groups, and users, audit ingestion and streaming, and org-wide policy enforcement like IP allowlisting.

### Dual-path JWT delivery

Access tokens arrive two ways: an `Authorization: Bearer` header for programmatic clients, or an encrypted `accessToken` cookie for the SPA. The cookie path isn't just `httpOnly` — the JWT itself is encrypted with a custom AES-256-GCM cipher before it's set (`crypto.js:10-48`), with a randomly generated IV prepended to the ciphertext each time. `httpOnly` already blocks JS access; the encryption is defense-in-depth against the token leaking through server-side channels like log files or request dumps.

<Callout type="warning">
The encryption key itself is derived with `crypto.createHash('sha256').update(secret).digest()` — a raw SHA-256 hash of `JWT_SECRET`, not PBKDF2, scrypt, or HKDF. It's also static, derived once rather than rotated. (`crypto.js:7`)
</Callout>

### Authorization: DENY beats ALLOW

`permission.service.js:236-248` evaluates deny policies before allow policies. Any matching DENY blocks the request outright, regardless of what ALLOW policies would otherwise grant — the same precedence model AWS IAM uses. It gives you a reliable "hard stop": a deny policy on a sensitive action holds even if a broader, later-added allow policy would otherwise open it up.

The actual check, in order:

1. Load the user's full access context — direct roles, group-derived roles, all attached policies — via `getCachedUserAccessContext()`, Redis-cached with version-busting through a `user:permissions:version` key.
2. SuperAdmin short-circuits to `{ allowed: true }`.
3. Split policies into DENY and ALLOW sets.
4. Any DENY match → blocked.
5. Any ALLOW match → granted.
6. No match → denied by default.

### Fail-open risk scoring

`riskEngine.js:16-40` calls the security-engine with a 500ms timeout. Any failure — timeout, network error, HTTP error — returns a neutral fallback: `{ risk_score: 0.1, is_anomaly: false }`. The authenticate middleware adds a second ceiling on top: a 100ms `Promise.race` around the whole call (`authenticate.js:200-206`).

That's a deliberate choice, not an oversight: a downstream ML service going down should never block login. The risk score is advisory — it triggers step-up auth on anomalies, it doesn't gate the login itself. (More on how that score actually gets computed further down.)

### Refresh token rotation — without family tracking

Every refresh (`services/auth/tokenService.js:23-53`) validates the old refresh token, deletes the old session, and issues a new session with a new refresh token. Rotation limits how long a stolen refresh token stays useful, since each use invalidates the one before it.

<Callout type="warning">
What's missing is **refresh token family tracking** — RFC 6819's theft detection pattern. If a token is stolen and both the attacker and the legitimate user use it, rotation just means whichever one goes last wins. There's no signal that a reuse-after-rotation happened at all.
</Callout>

### Request lifecycle, start to finish

1. `extractToken()` pulls the token from the `Authorization` header or the encrypted cookie.
2. If it's an API token (`iam_` prefix) → bcrypt-compare against `ApiToken.tokenHash`, check scope against `SCOPE_MAP`, check expiry.
3. If it's a JWT → verify signature (RS256) → check the JTI blacklist → check the Redis user-profile cache (`user:profile:{userId}:{profileVersion}`, 300s TTL, DB fallback on miss) → validate the session, either from a 60s Redis cache (`session:valid:{sessionId}`) or a live `prisma.session.updateMany` → enforce org-level policy.
4. Fire the ML risk check in parallel, capped at 100ms.
5. An anomalous result returns 403 (`AUTH_012`) unless the request is itself a reauth attempt.

Step-up reauth (`requireReauth.js`) gates eleven sensitive actions — password changes, MFA setup/disable, session revocation, org settings changes, data export, and similar. It accepts either a valid `x-reauth-token` (10-minute window) or a fresh password/TOTP check, and on success issues a new reauth JWT back to the client.

Rate limiting runs through eight named `express-rate-limit` instances with different windows per surface — login at 50/15min, registration at 20/hour, password reset at 3/hour, general API at 1000/15min, and so on. CSRF protection uses `csrf-csrf`'s double-submit cookie pattern, keyed to the JWT's `sessionId` claim (or `IP + user-agent` as a fallback), but — worth flagging directly rather than burying it —

<Callout type="warning">
**every auth endpoint is CSRF-exempt.** Login, register, refresh, password reset, email verification, and all OAuth routes skip CSRF checks entirely and rely on rate limiting instead. (`app.js:123-136`) That's a real gap, mitigated but not closed.
</Callout>

### Where this breaks down

- Session validation on a cache miss triggers a live `prisma.session.updateMany` — under high traffic with a cold cache, this is a real bottleneck.
- The `/metrics` "internal IP" check just tests whether the IP string starts with a private-range prefix — trivially spoofable, especially with `trust proxy` enabled.
- Analytics queries use `prisma.$queryRawUnsafe()` with string-interpolated column names, bypassing Prisma's type safety.
- There's no `X-Request-Id` propagation across services, so tracing a single request across API → security-engine → database means correlating by timestamp.
- The dev-mode email service spins up a fresh Ethereal test account on every server start if SMTP isn't configured.

### Auth & Session Numbers

| Metric | Value |
|---|---|
| Access token expiry | 15 min |
| Refresh token expiry | 7 days |
| User profile cache TTL | 300s |
| Session validity cache TTL | 60s |
| Permissions context cache TTL | 300s |
| Risk-scoring timeout | 500ms (HTTP) / 100ms (race ceiling) |
| Reauth token window | 10 min |
| Password reset token expiry | 1 hour |
| Email verification expiry | 24 hours |
| Account lock | progressive — 5min at 3 failures, 15min at 5, 30min at 7 |
| bcrypt rounds | 12 |
| Audit log retention | 90 days |
| API body size limit | 1MB |

Git history backs this up as an iteratively hardened system rather than a single audit pass — several commits (`#142`, `b4921c1`, `3c5db43`) show CodeQL and SonarQube flagging CSRF patterns and the audit-log hash chain across multiple rounds, not one clean sweep.

**What I'd change:** derive the cookie-encryption key with HKDF instead of a raw hash, add refresh token family tracking, move analytics off raw SQL onto Prisma's typed aggregate API (or a materialized view), and propagate a request ID through the whole stack.

## The Database

PostgreSQL 17, accessed exclusively through Prisma. Thirteen models cover users, sessions, roles, policies, groups, audit logs, OAuth accounts, API tokens, org settings, notifications, and revoked JTIs.

**UUID primary keys everywhere.** No coordination needed across a distributed system, unguessable, consistent. The tradeoff is index size and worse sequential-scan performance versus auto-increment integers.

**Composite unique constraints on junction tables** (`UserRole`, `GroupRole`, `UserGroup`, `RolePolicy`) rather than relying on the auto-generated UUID alone — this pushes duplicate-assignment prevention down to the database and doubles as the natural key for upserts.

**Six indexes on `AuditLog`** — `userId+createdAt`, `action`, `category`, `result`, `ipAddress`, `createdAt` — because it's the most-queried table in the system. The cost is write amplification: every insert touches all six indexes.

**A standalone `RevokedToken` table**, indexed on `expiresAt`, rather than a Redis-only blacklist — JWT blacklist entries need to outlive Redis availability, and the TTL index makes cleanup a straightforward batch delete instead of a table scan.

<Callout type="warning">
Some gaps worth knowing about if you're querying this data at scale: the audit-log `search` parameter uses chained `ILIKE '%...%'` conditions with no index support — a full table scan on every text search. `metadata` is stored as `JSON`, not `JSONB`, so there's no GIN index and no fast querying on nested fields even though the analytics layer already casts into `metadata->>'risk_score'`. And `OrganizationSettings` is a literal singleton — `ensureOrganizationSettings()` grabs the first row with no `WHERE` clause, so there's no path to multi-tenancy without a schema change.
</Callout>

## Auth System

The full lifecycle sits on top of the API and database layers above: bcrypt(12) registration, progressive account locking, Redis-backed session validation, JWT rotation, TOTP MFA via `otplib`, one-time bcrypt-hashed backup codes, Google/GitHub OAuth2, SHA-256-hashed password reset tokens, and org-level IP allowlisting.

The OAuth state parameter is worth a closer look: it's built as `randomHex(16) + '.' + HMAC-SHA256(randomHex(16), OAUTH_STATE_SECRET)`, verified with `crypto.timingSafeEqual`. That's stateless — no database round-trip to validate a callback — while still being forgery-resistant, and the constant-time comparison closes off timing attacks against the HMAC check itself.

<Callout type="warning">
OAuth access tokens are stored in plaintext in `OAuthAccount.accessToken` — if a provider's token grants API access beyond login, that's readable straight out of the database. Session `deviceInfo` is also just the raw user-agent string, with no fingerprinting or platform parsing behind it.
</Callout>

## The Security Engine: ML Risk Scoring

This is the most interesting — and most honestly-limited — subsystem. It's a Python/FastAPI service running an Isolation Forest model for unsupervised anomaly detection on audit-log data, with MLflow for experiment tracking and model registry.

### Why Isolation Forest

`sklearn.ensemble.IsolationForest`, `contamination=0.05`, `random_state=42`. The alternatives — One-Class SVM, Local Outlier Factor, autoencoders, or a supervised classifier like XGBoost — all assume something Bastion doesn't have: labeled attack data. There's no ground truth for "this login was malicious." Isolation Forest sidesteps that by being unsupervised, scales linearly with sample count, handles high-dimensional categorical data reasonably well after one-hot encoding, and is fast at inference (`O(t log n)`).

### Four features, not six

The model trains on exactly four fields pulled from `AuditLog`: `action`, `category`, `result` (all one-hot encoded), and `duration` (median-imputed, standard-scaled).

<Callout type="warning">
The internal docs (`security-engine.md:21`) describe six features, including IP address and user agent. The code implements four. There's no IP-based signal, no user-agent parsing, and no temporal features (hour of day, day of week), despite all three being documented as if they exist. This is the single most important thing to know about the risk engine: it's currently scoring on request shape and duration alone, not on anything identity- or network-adjacent.
</Callout>

### Model integrity and inference

Every trained model gets a SHA-256 hash written to `model_hashes.json`; on load, that hash is checked before `joblib.load()` runs. Since `joblib`/`pickle` deserialization can execute arbitrary code, the hash check is what stands between a tampered model file and remote code execution.

**Training** runs daily via a Kubernetes CronJob hitting `POST /train` (or manually via curl): pull the last 90 days of audit data (capped at 50,000 rows) through SQLAlchemy, fit a fresh `ColumnTransformer` + `IsolationForest` pipeline, log everything to MLflow, `joblib.dump()` the model, hash it, and sync the active version from the MLflow registry.

**Inference** is a single-row round trip: the Node API posts `{ action, category, result, duration }` to `/analyze`, the pipeline transforms and scores it, the raw anomaly score gets normalized into `[0, 1]` via `risk_score = 1.0 - ((score + 0.5) / 1.0)` and clipped, and anything above `0.7` is flagged anomalous — which the auth middleware turns into a 403.

<Callout type="warning">
No holdout validation exists anywhere in this pipeline. `train()` fits on 100% of the data — no train/test split, no cross-validation, nothing beyond a raw `mean_anomaly_score`. Every model gets deployed with zero measurement of precision, recall, or accuracy. There's currently no way to detect model degradation over time except noticing something feels off in practice.
</Callout>

A handful of smaller but concrete gaps: the training CronJob calls `/train` with no `X-Api-Key` header, so daily retraining silently 401s the moment auth is turned on for the security engine. `main.py` defines `API_KEY_HEADER`, `SECURITY_ENGINE_API_KEY`, and `verify_api_key()` twice — the first set is dead code. A `rate_limit()` dependency exists but is never wired to an endpoint via `Depends()`. Training runs synchronously inside an async FastAPI app, blocking the event loop for the 10-30 seconds a 50k-row fit with one-hot encoding takes. And the `0.7` anomaly threshold is hardcoded, not exposed as an env var.

**What I'd change:** implement the features that are already documented as existing (IP, user agent, path, temporal), add a real holdout split with metrics logged to MLflow, fix the training CronJob's auth, make training async or move it to a background queue, and delete the dead duplicate code in `main.py`.

## Frontend / Dashboard

React 18 + Vite + Tailwind, roughly eight top-level pages (login, register, dashboard, audit logs, audit stats, settings, security, user/RBAC management) built from about 30 page-level components plus a shared UI primitive package. Live audit data streams over SSE rather than WebSocket — a deliberate choice, since the data flow is strictly server-to-client and SSE gets you a native `EventSource` API with no handshake overhead, through plain HTTP.

Theming is CSS custom properties (`var(--ds-color-*)`) rather than Tailwind's `dark:` variant or CSS-in-JS, chosen so runtime theme switching doesn't need JS and the tokens can map directly from Figma exports. The migration is incomplete: some components still hardcode hex values (`StatsCards.tsx`, `SecurityAlerts.tsx`) while others use the custom properties (`UserInfoCard.tsx`). There are also two parallel settings implementations in the codebase — a 1,600-line state-based tab system in `SettingsPage.tsx` alongside a separate `settings/tabs/*` structure that duplicates it — and at least one component (`RoleDetail.tsx`) that calls the API directly instead of going through the shared service layer, bypassing the interceptor and CSRF handling everything else gets for free.

## Kubernetes Platform

A single-node K3s cluster on EC2, six services in the `aegismesh` namespace (backend, frontend, postgres, redis, security-engine, datadog-agent), zero-trust networking, nginx-ingress with MetalLB and cert-manager, and ArgoCD for GitOps delivery.

### Default-deny, then explicit allows

`default-deny.yaml` blocks all ingress and egress for every pod in the namespace; individual `allow-*.yaml` policies open exactly what's needed:

```
default-deny (all ingress+egress blocked)
├── allow-dns
├── allow-backend-to-postgres
├── security-engine-policies
├── allow-datadog-agent-traffic
├── allow-prometheus-scraping
├── allow-postgres-from-monitoring
├── allow-crowdsec-egress
├── allow-falcosidekick-egress
└── allow-ecr-refresher-traffic
```

<Callout type="warning">
`frontend` and `redis` have no explicit allow rules beyond DNS. Under a working CNI with default-deny enforced, the frontend can't reach the backend on port 5000, and redis is unreachable from the backend — either there's a gap in the policy set, or these are relying on something not visible in the manifests.
</Callout>

Secrets go into Git as Bitnami SealedSecrets rather than SOPS+age or Vault — no external KMS dependency, `kubeseal` encrypts directly to a `SealedSecret` CRD, decryptable only by the in-cluster controller (which means the controller's key itself is the thing to back up). Canary deployments go through Argo Rollouts on a fixed schedule — 20% traffic for 60 seconds, 50% for 60 seconds, then 100% — with no metrics-based analysis gate: simple to reason about, but no automated rollback if the canary starts erroring.

A few operational rough edges: the HPA targets the backend Deployment while Argo Rollouts manages its own replica count on the Rollout, so those two controllers fighting over replica count will cause flapping under load. The rollout itself starts at a single replica, so a canary that crashes takes down 100% of traffic on that one pod. No PodDisruptionBudgets exist, so a node drain takes out every pod simultaneously. Grafana still runs on `admin/admin`. And Terraform in this repo only manages two ECR repositories — there's no VPC, EKS, IAM, or RDS under Terraform at all; that infrastructure exists outside the repo.

## CI/CD

GitHub Actions handles CI — lint, test, build across all three apps on push/PR to `main`. On success, a separate CD workflow patches the Kustomize overlay image tags and opens an auto-merge PR from a `bot/deploy-<SHA>` branch. ArgoCD picks up the merge and reconciles the cluster. Every deployment is therefore a Git commit — full audit trail, review gate built in, no direct `kubectl set image` shortcuts.

Turborepo handles the monorepo's build graph, with remote caching in CI.

<Callout type="warning">
The CD workflow only patches backend and frontend images — the security-engine image stays pinned to `latest` in the overlay and is never updated by CI. Separately, the "already deployed" skip check does a plain substring match on the commit SHA against the patch files, so a short SHA that happens to be a substring of a longer one could produce a false skip.
</Callout>

Smoke tests are a curl check that the frontend responds and that rollouts succeed — no API-level health check, no canary verification. SonarQube scanning exists (`sonar-project.properties` is configured) but isn't wired into the CI pipeline at all — it only runs locally. Dependabot groups all patch and minor npm updates into a single PR, convenient but means an incompatible dependency bump can hide inside a batch.

## Observability

Four tiers run simultaneously: Datadog (APM traces, logs, profiling), Prometheus (15s scrape interval, two alert rules), Grafana (a single pre-built "AegisMesh Global Intelligence" dashboard, 14 panels across five sections), and Loki for log aggregation, plus MLflow for the ML side.

Running Datadog and the OSS stack in parallel is deliberate — Datadog gives distributed tracing and profiling out of the box, while Prometheus/Grafana/Loki give local-dev parity and a fallback if Datadog access lapses (the `DD_API_KEY` in the repo is a placeholder). The tradeoff is double instrumentation: both the backend and security-engine carry `dd-trace` and Prometheus client libraries side by side.

Alerting is thin: just two Prometheus rules (CPU over 0.5 cores, memory over 200MB), nothing log-based through Loki yet.

## Code Quality & the Custom Scanner

Code quality runs through three layers: local SonarQube scanning (Docker, no persistence between runs, so no historical trend data), GitHub CodeQL, and a custom 790-line Node.js scanner that checks for project-specific anti-patterns — stray `console.log`s, `0.0.0.0` binds, bare `except` clauses, `:latest` image tags, missing `USER` directives, missing K8s resource limits, cognitive complexity over 15. The custom scanner exists specifically because generic SAST tools don't catch things like "this Deployment has no resource limits." There's also a Python auto-fixer for around 20 common TypeScript issues (`==` → `===`, `||` → `??`, index-based array access → `.at(-1)`) — worth flagging that its `||`→`??` rewrite is a regex transform with no semantic verification, and that specific transform does change behavior around empty strings and `0`.

## What I'd Do Differently, System-Wide

1. Close the docs/code gap on the security engine's features — either build the missing IP/user-agent/temporal signals, or fix the docs to describe what's actually running.
2. Add holdout validation to the training pipeline — right now there's no way to know if a newly trained model is better or worse than the one it replaced.
3. Fix the HPA-vs-Rollout conflict before it causes replica flapping under real load.
4. Authenticate the training CronJob properly.
5. Add refresh token family tracking.
6. Replace the naive `/metrics` IP check with a real network-policy or service-mesh identity check.
7. Finish the CSS custom-property migration on the frontend.
8. Add e2e coverage — current tests touch under 10% of the frontend.
9. Resolve the `frontend`/`redis` NetworkPolicy gap, or document it as intentional if it is one.
10. Replace the regex-based Sonar auto-fixer with AST-based codemods (jscodeshift or ts-morph) so it can't silently change semantics.

<Callout type="note">
None of this is a takedown. It's the accurate state of a real system built solo, iterated on over ~129 PRs. The gaps above are exactly the kind of thing that separates "it works in the demo" from "I know precisely where this breaks and why." That second thing is the point of writing any of this down.
</Callout>
